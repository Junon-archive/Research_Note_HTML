핵심 통찰

시뮬레이터가 현대 HW를 충분히 정밀하게 모델링하지 못하면, 연구가 발견하는 "병목"과 "기회"가 실리콘에 존재하지 않는 허상일 수 있음.

핵심 기여 5가지

C1 Trace-driven Frontend

NVBit 기반 SASS 트레이서 → ISA-independent IR 변환. cuDNN 등 closed-source 라이브러리 시뮬레이션 최초 지원.

C2 GPGPU-Sim 4.0 Memory 상세화

sub-warp sectored coalescer, adaptive streaming L1, IPOLY L2 hashing, HBM dual-bus, write-validate sub-sector write 정책 도입.

C3 Validation 인프라

38개 microbenchmark + 자동 파라미터 tuner + correlation 시각화 도구로 구성된 검증 파이프라인 공개.

C4 Multi-generation 유연성

Kepler → Turing 4세대 GPU를 단일 프레임워크로 모델링하는 유연성 실증.

C5 Case Study

구식 베이스라인이 숨기는 두 가지 연구 기회(L1 throughput 병목 소멸, FR-FCFS 효과 극대화) 를 드러냄.

2. 논문 핵심 사항

해결한 문제 · 핵심 아이디어 · 해결 방법 · Contribution

Study Note

2.1 해결한 문제

기능 문제
매년 새 mISA 등장 → functional model 유지 비용 과다. cuDNN처럼 PTX 표현 없는 hand-tuned SASS 바이너리는 기존 시뮬레이터에서 실행 불가한 경우 존재.
정확도 문제
Volta QV100에서 GPGPU-Sim 3.x의 cycle error 94%, correlation 0.87 → 어떤 아이디어가 효과적인지 잘못 판단하게 만드는 수준.

2.2 핵심 아이디어

고정된 functional model을 직접 구현하는 대신 Trace-driven 시뮬레이션을 채택한 프레임워크 Accel-Sim을 제안.

SASS 트레이스를 직접 입력으로 받아 시뮬레이션함으로써 functional model 의존도를 낮추고, performance model(GPGPU-Sim 4.0)의 정확도와 유연성을 높임.

Accuracy는 성능 모델의 상세도가 결정하므로, memory system을 중심으로 현대 GPU 아키텍처를 역공학하여 반영함. 즉, 실제 HW가 실행하는 코드를 그대로 재현함으로써 Baseline을 실제 HW에 일치시키는 데 중점.

2.3 해결 방법 — Ambidextrous Frontend

NVIDIA NVBit을 활용한 트레이스 생성기(Trace-driven SASS Frontend)와 PTX 기반 에뮬레이션 모드(GPGPU-Sim PTX Emulation) 두 가지 모드를 모두 지원하는 양손잡이형 프론트엔드를 구축. 마이크로벤치마크를 통해 L1/L2 캐시·메모리 시스템 파라미터를 자동으로 추론·구성하는 튜닝 인프라도 도입.

2.4 Contribution 요약

C1 — Framework

최신 CUDA 바이너리 및 폐쇄형 라이브러리(cuDNN 등) 시뮬레이션이 가능한 확장 가능 프레임워크 제공 → 기존 오픈소스 시뮬레이터 한계 극복.

C2 — Accuracy

Kepler → Turing 세대 모델링 및 성능 모델 정확도 대폭 향상 → Cycle error 79%p 감소.

C3 — Infra

Microbenchmarks + Tuner + Correlator로 구성된 HW 검증 자동화 도구 공개 → 새로운 HW 설계 빠른 검증 방법론 제시.

3. 내게 필요한 정보

논문에서 직접 발췌·정리한 구현 관련 세부 사항

추가 검토 필요

3.1 시뮬레이션 모드 두 가지

Our new frontend supports both vISA (PTX) execution-driven and mISA (SASS) trace-driven simulation

Execution-driven (PTX)

사용 시점: 데이터 값에 의존적인 복잡한 시뮬레이션이나 동적 제어 흐름을 정확히 봐야 할 때.

새로운 명령어(ISA) 기능을 추가하는 경우, GPGPU-Sim 4.0 내부 Functional Simulator에 에뮬레이션 코드를 직접 추가해야 함.

Trace-driven (SASS)

사용 시점: 성능 분석 속도를 높이거나 SASS(mISA) 수준 분석이 필요할 때.

mISA → ISA-independent IR로 변환 후 사용. 기능적 에뮬레이션을 생략하고 핵심 작업만 선별적으로 시뮬레이션 → 기존 대비 월등한 속도와 최신 GPU SASS 정밀 모사 가능.

3.2 새 유닛 추가 방법 (To add a new specialized unit)

To add a new specialized unit, the user declares the new unit in the configuration file and maps the machine ISA op codes that use this unit in the ISA def file, as described in Figure 1.

Step 1

Config 파일에 새 유닛 선언.

Step 2

ISA Def 파일에서 해당 유닛을 사용하는 machine ISA op code를 매핑.

Step 3 (조건부)

Execution-driven PTX 지원이 필요한 경우, 새 명령어의 기능을 에뮬레이션하는 코드도 직접 작성해야 함 (= GPGPU-Sim 4.0 Functional Simulator 수정).

내 경우 어떻게 되는가? 단순히 HW 구성(캐시 용량, 레이턴시 등) 변경이라면 Performance Model만 수정하면 되므로 에뮬레이터 수정 불필요. 반면, 새 명령어(ISA)를 도입하거나 기존 명령어 동작을 PTX 수준에서 재정의해야 한다면 GPGPU-Sim 4.0 Functional Simulator(에뮬레이터) 수정이 필수적.

3.3 마이크로벤치로 직접 측정 불가한 파라미터

For other parameters that cannot be directly determined by our microbenchmarks (such as warp scheduling, memory scheduling, the L2 cache interleaving granularity and the L2 cache hashing function), Accel-Sim simulates each possible combination of these four parameters on a set of memory benchmarks.

Warp/memory scheduling, L2 interleaving granularity, L2 hashing function 등은 GPU 핵심 제어 로직·하드웨어 배선 수준에서 결정되어 단일 테스트만으로 역공학이 매우 어려움.

탐색 기반 추정

직접 측정값 없이, 발생 가능한 combination을 시뮬레이터에서 미리 설정하고 시뮬레이션 수행.

상관관계 분석

시뮬레이션 결과값과 실제 HW 프로파일링 데이터(nvprof, nsight 등)를 비교.

최적 조합 선정

실제 HW 데이터와 가장 높은 correlation을 보이는 설정을 자동으로 찾아 해당 GPU 모델의 파라미터로 고정.

3.4 140개 Workload — 어디다 쓰는가?

예를 들어 LLaMA3-8B 디코드 일부를 돌린다면, 저 workload들을 쓰지 않고 직접 trace를 만들어서 사용하면 됨. HW 수정 부분에 대한 성능 평가만 해보고 싶을 때 선택적으로 활용 가능. 저자들이 만들어놓은 trace 자산이 있다는 정도로 이해하면 충분.

4. 오개념 & 재학습

잘못 이해했던 내용과 이번에 바로잡은 내용

개념 정정 완료
오개념 1

Trace는 HW가 바뀌면 다시 만들어야 한다?
→ 특정 workload에 대한 Trace를 한 번 생성해두면, 시뮬레이터 설정을 변경해 다양한 HW 구조에서의 성능을 비교·평가할 수 있음. Trace는 ISA-independent IR로 변환되어 Performance Model에 전달되므로, 프로그램 논리 흐름·메모리 접근 정보는 Trace에 고정, 처리 속도 계산은 Config 설정에 달려 있음.

오개념 2

HW를 마음대로 수정하고 Trace 그냥 써도 된다?
→ HW 설정 변경 시 시뮬레이터가 인식할 수 있도록 Config 파일 수정은 반드시 필요. Trace: "어떤 명령어들이 어떤 순서로 실행되는가(데이터 흐름)". Config: "그 명령어들을 처리하는 HW 자원이 얼마나 있는가". Accel-Sim은 고정된 Trace를 입력받고 수정된 Config 설정에 따라 명령어 완료 속도를 계산함.

오개념 3

ISA 자체가 변경되는 경우에는?
→ ISA Def 파일을 수정해야 함. 새로운 명령어가 어떤 실행 유닛에서 처리되어야 하는지 매핑 정보를 업데이트 필요.

핵심 요약 (2.2.4) HW 설정 변경 → Trace 재사용 가능 (단, Config 수정 필수). 새 ISA 도입 → ISA Def 파일까지 수정 필요.

5. 내 연구 적용 전략

HVRP 시뮬레이션에서 Accel-Sim을 어떻게 수정·활용할 것인가

추가 실험 필요

5.1 수정 범위 — Layer별 구분

Layer 1 · Config만 수정

  • Offloading 대상 메모리 계층(L1/L2/HBM) 설정
  • 용량, 레이턴시, 대역폭 파라미터 조정
  • 빠름, 검증 쉬움

Layer 2 · GPGPU-Sim 4.0 C++ 수정

  • Prefetch 정책 구현 — 어떤 head를 언제 prefetch할지 스케줄링 로직
  • 기존 구조에 없으므로 C++ 코드 레벨 수정 불가피
  • 연구의 핵심 contribution에 해당

Layer 3 · ISA Def 수정 (조건부)

  • Prefetch를 명시적 명령어로 넣으려면 필요
  • 기존 LD/ST 명령어로 모델링하면 불필요
  • 초기 단계에서는 아마 불필요

5.2 Trace 재사용 전략

재사용 가능한 경우

Prefetch 정책 변경, 메모리 계층 파라미터 변경 등 Config 레벨에서만 바꾸는 경우 Trace를 그대로 재사용하며 반복 실험 가능.

재생성 필요한 경우

Prefetch 동작 자체가 메모리 접근 패턴을 바꾼다면 — 즉, prefetch로 인해 실제로 실행되는 메모리 instruction이 달라진다면 — trace를 다시 생성해야 함. 이 경우 execution-driven mode로 fallback하거나, prefetch를 별도 inject하는 방식 고려 필요.

5.3 Validation 인프라 활용 계획

기존 활용

논문이 제공하는 38개 microbenchmark 중 memory latency/bandwidth 관련 항목들을 offloading 경로 검증에 바로 활용 가능.

Custom 추가

Prefetch 효과 측정을 위한 custom microbenchmark 추가가 필요할 수 있음. 논문 Figure 3 (L1 latency 벤치마크) 작성 패턴을 그대로 따르면 됨.

6. 나만의 결론

논문 정독 후 내 연구에서 결정해야 할 포인트

결정 요청
결론

Trace 고정 + HW 반복 실험 구조로 활용하되, prefetch가 instruction 흐름을 바꾸는 시점의 전환 기준을 사전에 설정해야 함

Accel-Sim은 Trace를 고정하고 HW를 바꿔가며 실험하기에 최적화된 도구. 내 연구에서도 trace를 한 번 생성해두고 offloading/prefetch 정책과 메모리 파라미터를 바꾸며 반복 실험하는 방식으로 활용.

HVRP 적용
결정 포인트 1 — 수정 범위

Prefetch를 명시적 명령어로 넣을지 vs 기존 LD/ST로 모델링할지. 후자라면 ISA Def 수정 불필요.

결정 포인트 2 — 모드 전환 기준

Prefetch가 instruction 흐름 자체를 바꾸는 시점에서 execution-driven으로 전환하는 기준을 사전에 설정해두어야 함.

실험 설계 트레이드오프

Trace는 특정 HW에서 실행된 결과물이므로, prefetch로 바뀌는 메모리 접근 패턴이 trace 자체에 반영되지 않음. 이 처리 방식이 실험 설계의 핵심 결정 포인트.

단계 1 — NVBit 1.7.6 빌드

설치 및 빌드

NVBit를 서버에 설치, 빌드 자체는 약 12초 만에 완료. ARCH=sm_89 타겟으로 tracer_tool.so(2.8MB)와 post-traces-processing(1.2MB) 빌드.

함정 — CUDA 버전

서버에는 CUDA 버전이 두 개 있는데, L40에서 NVBit을 사용하려면 반드시 버전 12.4 사용 필수. 기본으로 설정된 구버전(11.8)을 쓰면 설치 자체가 안 됨.

setup_accelsim.sh가 conda CUDA 11.8을 설정하므로, 이후 module load cuda/12.4로 override 필수. NVBit 1.7.6은 CUDA 11.8 nvcc로는 sm_89 빌드 불가 (sm_89는 CUDA ≥ 12.0 필요).

빌드 후 생성 파일

  • accel-sim-framework/util/tracer_nvbit/tracer_tool/tracer_tool.so (2.8MB)
  • accel-sim-framework/util/tracer_nvbit/tracer_tool/traces-processing/post-traces-processing (1.2MB)

단계 2 — LLaMA3-8B Baseline

NVBit 없이 모델이 정상 작동하는지 단독으로 확인하는 단계. LLaMA3-8B (Meta, fp16) single-token decode 실행.

  • fp16 모델 로드
  • greedy decode (max_new_tokens=1)
  • 사용 GPU 메모리: 16GB

결과: input "Hello" → output ",", generate time 0.655s, GPU VRAM 16.06GB.

단계 3 — NVBit 전체 커널 덤프

NVBit를 붙여 LLaMA3-8B 커널 확인, 총 1,430개의 커널 기록.

실패한 시도

  • Job 11204 (ACTIVE_FROM_START=1): 모델 로드 시 JIT 컴파일 단계에서 NVBit이 Ada SM 8.9의 CALL.REL.NOINC 명령어를 처리하지 못해 경고 메시지가 초당 수만 줄씩 폭주 → inference 도달 불가.
  • 외 다수.

해결책

ACTIVE_FROM_START=0 + Python 코드에서 warm-up forward를 1회 실행(JIT 완료)한 뒤 cudaProfilerStart() 호출. --nodelist=acas1로 노드 강제 지정.

결과

  • 소요: 약 42분, 노드: acas1
  • 총 1,430개 커널 trace (ctx_0x71aa850)
  • 파일: kernel-1400-ctx_0x71aa850.trace.xz ~ kernel-2501-ctx_0x71aa850.trace.xz
  • FlashAttention 커널 위치 확인: pytorch_flash::flash_fwd_kernel, ID 1450, 1493, …, 2783 (간격 43, 총 32개)

단계 4 — FlashAttention 커널 Trace

FlashAttention 커널만 골라내는 작업. 총 32개가 선택되어 각각 약 140KB 파일로 저장됨.

설계 오류

  • DYNAMIC_KERNEL_RANGE="1450-2783@.*pytorch_flash.*flash_fwd_kernel.*" 지정.
  • 그러나 ACTIVE_FROM_START=0 모드에서 NVBit의 내부 카운터(ctx_kernelid)는 cudaProfilerStart() 이후 1부터 재시작.
  • 단계 3의 ID(1450~2783)는 전체 프로세스 기준이므로 profiler 창 안에서는 무효 → 범위 필터가 무시되고 1,430개 전체가 trace됨 (약 80분).

사후 처리

  • 1,430개 중 flash_fwd_kernel에 해당하는 32개를 단계 3 kernelslist와 대조해 선별
  • traces/llama3_flash_attn/에 symlink 생성
  • post-traces-processing 실행
  • .traceg.xz 32개 생성

출력

  • kernel-{1450,1493,…,2783}-ctx_0x65ab690.traceg.xz 32개 (각 138~140KB)
  • kernelslist.g (1.2KB)

주의: 단계 3(ctx_0x71aa850)과 단계 4(ctx_0x65ab690)는 CUDA context 주소가 다름 (별도 잡 실행). 커널 순서의 결정성(determinism)은 검증하지 않음.

단계 6 — SM89_L40 gpgpusim.config 작성

시뮬레이터 config를 L40 하드웨어 스펙에 맞게 수정. SM 수(46 vs 142), L2 크기(4MB vs 48MB), 메모리 채널(16 vs 12), 클럭 등 주요 파라미터 불일치. → L40 공개 스펙을 기반으로 SM86_RTX3070을 베이스 삼아 SM89_L40 config를 직접 작성.

  • 베이스: configs/tested-cfgs/SM86_RTX3070/gpgpusim.config
  • 생성: configs/tested-cfgs/SM89_L40/gpgpusim.config + trace.config

수정 항목 요약

  • GPU 세대 번호: 86 → 89 (L40은 Compute Capability 8.9)
  • SM: 46개 → 142개
  • 메모리 채널 수: 16개 → 12개
  • 2차 캐시 크기: 4MB → 48MB
  • 클럭 속도: 1132MHz → 1110MHz

단계 7 — SM89_L40 Config 시뮬레이션

  • 파티션: cpu · 노드: acas1 · 소요: 103초
  • Config: SM89_L40/gpgpusim.config (수정한 config)
  • 결과 저장: sim_run/llama3_flash_attn_SM89/sim_output.txt

해석

  • IPC·사이클 차이 1% 미만: SM 수(46→142), L2 크기, 클럭을 L40에 맞게 수정했으나 워크로드가 작아(grid 1×1×32, 32 CTAs) SM 활용률 차이가 크지 않음.
  • L1D miss = 1.0000 유지: config 불일치가 원인이 아님. FlashAttention은 공유 메모리(SRAM) 중심의 타일 연산 알고리즘으로, global memory 접근이 L1D를 거치지 않고 직접 L2/DRAM으로 가도록 설계되어 있어 그런 것으로 파악됨. 이는 알고리즘 고유 특성.

주요 함정

  1. CALL.REL.NOINC: Ada SM 8.9 이상의 신규 분기 명령어. NVBit 1.7.x가 미처리 → 경고 폭주. ACTIVE_FROM_START=0 + warm-up으로 회피.
  2. ctx_kernelid 재시작: ACTIVE_FROM_START=0 시 cudaProfilerStart() 이후 ID 카운터가 1부터 재시작. DYNAMIC_KERNEL_RANGE의 ID 범위 지정이 무효가 되고, 이름 regex만 유효함.
  3. IPOLY 해싱 제약: GPGPU-Sim의 IPOLY hash function은 파티션 수 16/32/64만 지원. L40의 24파티션(12채널×2)은 미지원 → memory_partition_indexing 0으로 우회.

세션별 요약

세션 1 — End-to-end 파이프라인 완주 (단계 1~5)

수행: LLaMA3-8B (Meta, fp16) single-token decode에서 pytorch_flash::flash_fwd_kernel 커널 32개를 NVBit 1.7.6으로 trace하고, Accel-Sim 4.2.0 trace-driven 시뮬레이션 파이프라인을 end-to-end로 통과.

의미: 실제 하드웨어(L40 Ada SM 8.9) trace를 가상 하드웨어(A100 Ampere SM 8.0) config로 시뮬레이션하는 파이프라인이 정상 작동함을 검증.

세션 2 (2026-06-11) — SM89_L40 config 작성 및 재시뮬레이션 (단계 6~7)

수행: L40 공개 스펙을 기반으로 SM86_RTX3070 베이스에서 SM89_L40 gpgpusim.config를 작성하고, 동일 trace(flash_fwd_kernel 32개)를 재시뮬레이션해 세션 1 결과와 비교.

수정 파라미터: compute capability, SM 수(46→142), 메모리 채널(16→12), L2 크기(4MB→48MB), 클럭(1132→1110MHz / 3500→2000MHz).

결과 요약: IPC 차이 0.6%, 사이클 차이 0.6%로 사실상 동일. L1D miss = 1.0000 유지.

Key핵심 결론

  • 실제 하드웨어(L40 Ada SM 8.9) trace를 가상 하드웨어 config로 돌리는 NVBit → Accel-Sim end-to-end 파이프라인이 정상 작동함을 검증.
  • L1D miss = 1.0000은 config 불일치가 원인이 아니라, FlashAttention의 tiled SRAM 알고리즘이 global memory 접근 시 L1D를 bypass하도록 동작하는 알고리즘 고유 특성으로 판단됨. (-gpgpu_gmem_skip_L1D 0 설정에도 불구하고 공유 메모리를 통한 데이터 재사용이 지배적)

Open미확인 · 미검증 사항

  • L1D miss = 1.0000 정밀 검증: 알고리즘 특성으로 추정했으나, 확정하려면 non-FlashAttention 커널(layernorm, embedding 등)과의 L1D miss rate 비교 필요.
  • Instruction semantics 매핑 정확도: Ada SM 8.9 trace를 SM 8.0 config로 돌렸을 때 GPGPU-Sim이 ISA를 올바르게 매핑하는지 미확인.
  • 커널 순서 결정성: 단계 3(ctx_0x71aa850)과 단계 4(ctx_0x65ab690)의 커널 launch 순서가 동일한지 미검증.
  • SM89_L40 config 한계: 공식 GPGPU-Sim 검증(RTL 비교) 미수행으로 1차 근사 수준. DRAM 타이밍(GDDR6X→GDDR6 차이)과 파이프라인 세부 구조는 미수정.

Next향후 방향

  • 다른 커널 유형(embedding lookup, layernorm, matmul)에 동일 파이프라인을 적용.
  • MoH head-wise offloading 시뮬레이션을 위한 baseline 확보 완료 → 이를 토대로 다음 단계 진행.

왜 Accel-Sim인가?

GPGPU-Sim 3.x의 방식

PTX(가상 ISA, vISA) 기반의 실행-구동(execution-driven) 방식을 사용함.

문제점

  • PTX는 컴파일러 중간 표현이지 실제 HW가 실행하는 코드가 아님.
    • 실제 GPU는 SASS를 실행.
    • PTX → SASS 컴파일 과정에서 레지스터 할당·스케줄링이 달라지므로 시뮬레이션 결과가 실리콘과 불일치.
  • cuBLAS, cuDNN, PyTorch 같은 라이브러리는 PTX를 노출하지 않음.
    • 바이너리로만 배포되기 때문에 PTX 기반 시뮬레이터로 실행 불가.
  • Execution-driven 방식은 GPU 기능 모델(functional model)이 반드시 필요.
    • 새 아키텍처가 나올 때마다 기능 모델부터 다시 구현해야 함을 의미.

의문 — Trace-driven도 단점이 있지 않을까?

맞음. 실제 HW 동작과 시뮬레이터 동작이 달라지면 trace가 틀려짐. 자세한 내용은 어펜딕스 참조.

Trace-Driven 방식

실제 GPU (L40, V100 등)
        │
        │  NVBit (binary instrumentation)
        ▼
   SASS 명령어 스트림 기록 (traces/)
        │
        │  파일로 저장
        ▼
   GPGPU-Sim 4.x (소프트웨어 CPU 시뮬레이터)
        │
        │  trace를 읽어서 사이클 단위로 재생
        ▼
   성능 통계 (IPC, cache miss rate, memory BW, ...)

Accel-Sim의 장점

  • 기능 모델 불필요. trace에는 이미 실행 결과(레지스터 값, 메모리 접근 주소)가 들어 있으므로 시뮬레이터가 연산을 직접 수행할 필요 없음.
  • cuBLAS, cuDNN, PyTorch를 포함해 CUDA 바이너리로 실행 가능한 모든 것을 추적 가능. 즉, 소스 코드 불필요.
  • 재현성. trace를 한 번 찍어두면 HW 없이도 반복 시뮬레이션 가능. 파라미터를 바꿔가며 실험할 때 매우 유리.

Accel-Sim 4개의 구성요소

Accel-Sim은 4개의 주요 컴포넌트로 구성.

1. Accel-Sim Tracer

  • NVBit 기반의 binary instrumentation 도구.
  • 실제 GPU에서 CUDA 바이너리를 실행하면서 SASS 명령어 스트림을 파일로 기록.
  • 출력: kernelslist.g (커널 목록) + 각 커널별 trace 파일.

2. Accel-Sim SASS Frontend + Simulation Engine

  • GPGPU-Sim 4.0의 성능 모델 위에 exec_·trace_ 접두사를 가진 새 클래스를 상속해 구현.
  • exec-driven 모드에서는 기능 모델에서, trace-driven 모드에서는 trace 파일에서 다음 명령어를 공급.
  • GPGPU-Sim의 성능 모델(SM 구조, warp scheduler, 메모리 계층 등)은 그대로 유지하면서 "명령어 공급" 방식만 교체한 구조.

3. Accel-Sim Correlator

  • 성능 모델에서 나온 통계와 실제 HW 프로파일링 결과를 매칭·플롯·비교하는 도구.

4. Accel-Sim Tuner

  • 마이크로벤치마크 스위트를 자동으로 실행해 GPGPU-Sim 설정 파일을 자동 생성하는 도구.
  • HW 스펙 헤더 파일을 제공하면 마이크로벤치마크를 돌려 캐시 크기, 레이턴시, BW 등을 측정하고 그 결과를 config 파일로 출력.

GPGPU-Sim 4.x와의 관계 — 헷갈리는 부분

표 1. GPGPU-Sim 3.x / 4.x / Accel-Sim Framework 비교
항목설명
GPGPU-Sim 3.x 기존 시뮬레이터. PTX 기반, execution-driven. UBC 그룹 유지보수.

Accel-Sim Framework GPGPU-Sim 4.x를 내부적으로 활용. 그 위에 Tracer / Frontend / Correlator / Tuner를 얹은 통합 프레임워크.

./gpu-simulator/setup_environment.sh 실행 시 적합한 버전의 GPGPU-Sim 4.x가 자동으로 클론되고 Accel-Sim과 통합됨. GPGPU-Sim을 별도로 받을 필요 없음.

지원 아키텍처

  • 현재 ISA 정의 파일은 Kepler, Pascal, Turing, Volta 세대 대상.
  • 공식적으로 검증된(tested) config는 주로 SM7_QV100 (Volta, V100) 기준.
  • config 파라미터(캐시 크기, BW 등) 튜닝은 별도 필요.
  • L40 기준으로 실험하려면 SM89 config를 직접 작성해야 함.
  • 다른 아키텍처가 필요하면 config를 수정해서 만드는 방식.

Key정확도 및 신뢰도

ISCA 2020 논문 기준, 제작자가 명시한 내용.

  • Cycle-accurate가 아님 — RTL 시뮬레이터가 아니므로 클럭 단위 정확도 없음.
  • Trend-accurate — IPC, cache hit rate, memory bandwidth utilization 같은 성능 트렌드는 잘 포착.
  • 검증된 workload 범위 내에서는 상당히 신뢰 가능한 수준.
  • 직접 작성한 새 아키텍처 config는 튜닝 전까지 오차 큼.

Open한계

  • 실제 GPU가 tracing에 필요 — trace 생성 단계는 실리콘 필수. 시뮬레이션 재생은 CPU만으로 가능.
  • NVBit가 지원하는 아키텍처에 제한 — sm_89까지는 현실적, Blackwell(sm_120)은 불확실.
  • Closed-source 라이브러리의 주소 분석 어려움 — trace 자체는 추출 가능하나 semantic 이해는 별도.
  • Config 파일의 품질이 결과를 결정 — 잘못 쓴 config는 완전히 다른 결론으로 이어짐.
  • 멀티 GPU 시뮬레이션은 아직 로드맵 단계.

어펜딕스 — Trace-driven이 문제가 되는 경우

Branch divergence 처리가 달라질 때

warp 내에서 일부 thread가 다른 branch를 탔을 때, 시뮬레이터가 다른 재합류(reconvergence) 결정을 내리면 실제 실행 순서와 달라질 수 있음. trace는 이를 반영하지 못함.

메모리 동작에 의존적인 제어 흐름이 있을 때

Atomic 연산, 동기화 barrier 결과에 따라 다음 명령어가 달라지는 구조라면, trace가 특정 HW 동작을 전제로 찍혀 있어서 다른 마이크로아키텍처로 재생 시 의미가 달라질 수 있음.

Spinlock, polling loop

Accel-Sim README에도 spinlock handling 옵션이 별도로 있는 이유. 루프 반복 횟수가 HW 상태에 따라 달라지는데, trace는 특정 HW에서 실제로 돈 횟수만 기록.

한 줄 요약: trace-driven은 "명령어 공급"과 "마이크로아키텍처 결정"이 분리되어 있어서, 마이크로아키텍처가 명령어 흐름 자체에 영향을 미치는 상황에서는 부정확.

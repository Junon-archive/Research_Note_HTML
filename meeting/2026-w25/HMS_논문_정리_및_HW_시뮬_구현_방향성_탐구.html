1. 논문 핵심 사항

GPU 메모리 용량 문제에서 SCM, DRAM Cache, Bypass Policy, CTC, AMIL로 이어지는 문제-해결 흐름 정리임

Narrative

1.1 해결하고자 한 핵심 문제

  • GPU의 메모리 용량 문제
  • UVM, Multi-GPU를 사용해도 한계가 있음

1.2 핵심 아이디어

핵심 문제에 대한 해결책: [SCM] — DRAM의 속도와 SSD의 용량의 이점을 가진 중간 단계 HW

  • SCM을 메모리로 사용
  • 파생되는 문제 A: 긴 쓰기 지연 시간, 높은 에너지 소비 해결 필요

문제 A에 대한 해결책 · DRAM Cache

  • 이때 DRAM으로 Cache를 만들어 사용
  • 파생되는 문제 B-1: DRAM Cache가 쉽게 thrashing 상태가 되어 오히려 성능을 저하시키고 대역폭을 낭비하게 됨
  • 파생되는 문제 B-2: DRAM Cache 히트 여부 확인을 위해 매번 외부 DRAM에 태그를 조회해야 함
  • 추가적인 태그 조회 트래픽이 실제 데이터 요청과 대역폭 경쟁을 벌여 전체적인 대역폭 효율 저하 발생

문제 B/C에 대한 해결책

Bypass Policy
SCM-aware DRAM Cache Bypass Policy 도입, 성능 기여도가 낮은 데이터를 DRAM에 캐싱하지 않고 바로 SCM으로 보냄
CTC
Configurable Tag Cache. L2 Cache의 일부 공간을 태그 캐싱 용도로 재구성(repurpose)하여 온칩(on-chip)에서 빠르게 태그 조회
문제 C
CTC 미스 발생 가능. 이때 태그를 조회하기 위해 여러 번의 DRAM 접근 필요
AMIL
한 DRAM 로우(Row) 내의 모든 태그와 메타데이터를 마지막 컬럼(Last-column)에 집중적으로 배치
AMIL의 직접 효과단 한 번의 컬럼 접근으로 해당 로우의 모든 태그를 가져올 수 있게 함

1.3 Contribution이 뭔지

GPU 맞춤형 SCM-aware DRAM 캐시 설계

  • SCM의 물리적 특성인 긴 쓰기 지연 시간 및 높은 에너지 고려
  • 워크로드의 접근 패턴인 공간 지역성, 접근 빈도 통합 고려
  • 지능형 바이패스 정책(SCM-aware Bypass Policy) 최초 제안

태그 조회 효율 극대화

  • Configurable Tag Cache (CTC)
  • Aggregated Metadata-In-Last-column (AMIL)

성능 및 에너지 효율성 입증

  • 기존의 HBM 기반 GPU 대비 성능 최대 12.5배, 평균 2.9배 향상
  • 에너지 소비 최대 89.3%, 평균 48.1% 절감

4.1 수정 방향성

구조 자체는 Accel-sim/GPGPU-sim 그대로 재활용하고, 필요한 추가를 주입하는 방식으로 접근하면 됨

Layer 1 — Config 수정만으로 끝나는 부분

  • Predictor SRAM 용량·레이턴시 파라미터 설정
  • 새 캐시를 기존 L1/L2 계층과 다른 레이턴시로 설정
  • HBM 대역폭 파라미터 설정
  • KV 접근 baseline 설정
  • .config 파일 수정으로 처리 가능

Layer 2 — GPGPU-Sim C++ 코드 수정

  • 핵심 기여에 해당하는 수정 범위
  • Predictor SRAM Controller 신규 작성
  • KV Prefetch Scheduler 구현
  • HMS의 HMSController.cc에 대응되는 파일군으로 이해 가능

Layer 3 — ISA Def 수정

  • 기존 LD/ST로 KV 접근을 모델링하는 한 초기 단계에서 불필요
  • Prefetch를 memory controller가 투명하게 처리하면 기존 SASS trace 그대로 사용 가능

Layer 2 세부 내용

1. Predictor SRAM Controller

Markov 테이블 조회 로직 신규 작성. enqueue()를 받아서 head_id → table hit/miss 판별 → tick에서 결과 반환하는 구조임

2. KV Prefetch Scheduler

gmmu.cc 패턴 참고. 테이블 히트 시 다음 토큰의 KV를 미리 HBM에서 fetch하는 구조임

HMS의 PCIE_VALIDATE 큐 패턴을 여기서 재활용할 수 있음 — CPU-resident 데이터 fetch 경로가 이미 구현되어 있으니, 분기 조건만 head_id 기반으로 추가하면 됨

4.2 [핵심] 구현의 방향성을 잡아보자

구현할 내용 크게 세 개임

대상평가 내용
Predictor cache용량, lookup latency, hit rate, DRAM fallback, 면적, 전력

4.2.1 Head마다 달라지는 kv 이동을 시뮬레이터에서 어떻게 표현?

ISA 수정 없이, 메모리 컨트롤러 레벨에서 mask로 처리하는 방식 생각중

구체적으로:

  • KV cache에서 선택된 8개(K=8일 때 가정) head의 데이터에만 유효성 비트를 세우고, 나머지 8개 head의 fetch 요청을 컨트롤러 레벨에서 suppress
  • 이렇게 하면 SASS trace에 있는 기존 LD 명령어를 그대로 쓰면서, memory controller가 실제로 HBM에 나가는 트래픽만 줄이는 효과를 만들 수 있지 않을까 싶음

4.2.2 Predictor SRAM을 시뮬레이터에서 어떻게 모델링할 거냐?

  • 기존 L1 캐시 구조를 별도로 하나 더 만들고
  • 용량·레이턴시·포트 수만 SRAM 스펙에 맞게 Config에서 조정하는 방식 생각중

구체적으로:

  • Accel-sim의 캐시 구조는 용량, associativity, hit/miss latency를 config 파라미터로 받음
  • 여기에 “Predictor SRAM”이라는 이름의 새 캐시 인스턴스를 추가
  • 레이턴시는 SRAM 면적/전력을 먼저 추정하고, 거기서 나온 access latency를 config에 주입함 (어케 추정함??) (확인 필요)
  • 테이블 lookup 요청이 이 SRAM 인스턴스에 먼저 들어오고, miss 시에만 DRAM으로 fallback하는 경로를 만들기
  • DRAM fallback은 실제로는 발생 안 한다고 가정 — 테이블이 SRAM에 완전히 올라간다고 가정

4.2.3 Markov 테이블 조회와 KV Prefetch를 어떻게 파이프라인으로 연결할 거냐?

조회 결과가 나오기 전에 prefetch를 언제 트리거하냐, 레이턴시 겹침을 어떻게 모델링할 것인가?

  • HMS의 “판단 → 큐잉 → tick 실행” 패턴을 그대로 써도 될 듯
  • 조회 요청을 큐에 넣는 순간 prefetch 큐도 동시에 세우는 방식 생각중

구체적으로:

  • 토큰 t의 attention 시작 시점에 SRAM 조회 요청을 enqueue
  • SRAM hit이 확인되는 순간(hit latency 후) KV prefetch 큐에 해당 head들의 HBM 주소를 넣기
  • miss 시에는 원본 라우터 fallback 경로 — 이건 기존 HBM 접근 경로 그대로

Glossary

5.1 용어 정리

용어의미 (Description)관련 기술
SCMStorage-Class Memory: DRAM과 Flash 사이의 특성을 가진 비휘발성 메모리. 예: PCM시스템 기반
HMSHeterogeneous Memory Stack: DRAM과 SCM을 3D 적층 구조로 결합한 메모리 조직시스템 구조
TADTag-And-Data: 태그와 데이터를 함께 저장하는 전통적인 DRAM 캐시 조직 방식AMIL 비교군
CTCConfigurable Tag Cache: L2 캐시의 일부를 DRAM 캐시 태그 저장소로 전환하는 기법태그 조회 최적화
AMILAggregated Metadata-In-Last-column: 로우 내 마지막 컬럼에 메타데이터를 집중 배치하는 구조태그 조회 최적화

Paper Overview
Main Claim

SCM으로 GPU 메모리 용량 벽을 넘되, DRAM cache의 대역폭 오버헤드를 줄이는 것이 HMS의 핵심임.

AMIL, CTC, SCM-aware bypass policy를 통해 oversubscribed HBM 대비 최대 12.5×, 평균 2.9× 성능 향상을 달성함.

HPCA 2024 DRAM Cache SCM
제목

Bandwidth-Effective DRAM Cache for GPU s with Storage-Class Memory - HMS (이하 HMS)임.

출처

HPCA 2024 / Jeongmin Hong, Sungjun Cho, Geonwoo Park, Wonhyuk Yang, Young-Ho Gong, Gwangsun Kim / POSTECH & Soongsil University임.
DOI: 10.1109/HPCA57654.2024.00021임.

한 줄 직관

“SCM은 sequential read엔 HBM만큼 빠르지만 random write엔 치명적임 — 이 비대칭성을 3차원 score로 정량화해 DRAM cache가 정말 필요한 데이터만 골라 담으면, oversubscribed HBM보다 훨씬 낫다”는 직관임.

분류 태그

#DRAM-cache #SCM #PCM #GPU-memory #HBM #HW-SW-codesign #memory-oversubscription #accelerator-simulation

핵심 기여

  1. HMS (Heterogeneous Memory Stack): SCM die + DRAM die를 TSV로 3D stacking, 같은 channel에서 shared bus로 유연한 BW 배분함.
  2. AMIL (Aggregated Metadata-In-Last-Column): DRAM cache row의 모든 tag를 마지막 column의 32 B data portion에 집약 → single column access로 row 전체 tag 취득, ECC 온전히 유지함.
  3. SCM-aware DRAM Cache Bypass Policy: (spatial locality + write intensity → SCM penalty score) × access frequency → DRAM-affinity score의 2-level 비교 구조임.
  4. CTC (Configurable Tag Cache): L2 cache way의 일부를 tag cache로 repurpose, user-configurable 분할함.
  5. SCM power/mode management: throttling(timing parameter 배가) + SLC/MLC/TLC 모드 전환으로 footprint 크기에 adaptive 대응함.

1. 논문 핵심 사항

GPU 메모리 용량 문제에서 SCM, DRAM Cache, Bypass Policy, CTC, AMIL로 이어지는 문제-해결 흐름 정리임.

STEP 1핵심 문제

STEP 2핵심 해결책 · SCM

DRAM의 속도와 SSD의 용량의 이점을 가진 중간 단계 HW인 SCM을 메모리로 사용함.

STEP 3파생 문제 A

SCM 사용 시 긴 쓰기 지연 시간, 높은 에너지 소비를 해결해야 됨.

STEP 4DRAM Cache

문제 A의 해결책으로 DRAM을 Cache로 만들어 사용함.

STEP 5파생 문제 B

DRAM Cache thrashing과 외부 DRAM 태그 조회 트래픽이 대역폭 효율을 떨어뜨림.

문제 B-1에 대한 해결책 · Bypass Policy

  • SCM-aware DRAM Cache Bypass Policy를 도입함.
  • 성능 기여도가 낮은 데이터를 DRAM에 캐싱하지 않고 바로 SCM으로 보냄.

문제 B-2에 대한 해결책 · CTC

  • Configurable Tag Cache (CTC)를 사용함.
  • L2 Cache의 일부 공간을 태그 캐싱 용도로 재구성(repurpose)하여 온칩(on-chip)에서 빠르게 태그를 조회함.
  • 파생되는 문제 C는 CTC 미스 발생 가능성임.
  • CTC 미스 발생 시 태그를 조회하기 위해 여러 번의 DRAM 접근이 필요함.

문제 C에 대한 해결책 · AMIL

  • Aggregated Metadata-In-Last-column (AMIL)을 사용함.
  • 한 DRAM 로우(Row) 내의 모든 태그와 메타데이터를 마지막 컬럼(Last-column)에 집중적으로 배치함.
  • 단 한 번의 컬럼 접근으로 해당 로우의 모든 태그를 가져올 수 있게 함.

1.3 Contribution이 뭔지

  • GPU 맞춤형 SCM-aware DRAM 캐시 설계임.
    • SCM의 물리적 특성(긴 쓰기 지연 시간 및 높은 에너지)과 워크로드의 접근 패턴(공간 지역성, 접근 빈도)을 통합적으로 고려한 지능형 바이패스 정책(SCM-aware Bypass Policy)을 최초로 제안함.
  • 태그 조회 효율 극대화함.
    • Configurable Tag Cache (CTC) 적용함.
    • Aggregated Metadata-In-Last-column (AMIL) 적용함.
  • 위 사항들을 도입하여 성능 및 에너지 효율성 입증함.
    • 기존의 HBM 기반 GPU 대비 성능을 최대 12.5배(평균 2.9배) 향상함.
    • 에너지 소비를 최대 89.3%(평균 48.1%) 절감함.

Actionable Pattern
Implementation Pattern

“판단 → 큐잉 → tick 실행” 패턴을 Markov prefetch scheduler에도 그대로 적용 가능함.

post_process_cache_miss() → fill_pending queue → tick() 흐름을 재활용하는 방향임.

Queue-based Scheduler

큐에 넣는 단계임.

실행

4.1 수정 방향성

구조 자체는 Accel-sim/GPGPU-sim 그대로 재활용하고, 필요한 추가로 주입하는 방식으로 접근하면 됨.

수정 범위 요약

Layer수정 내용
Layer 1 — Config 수정만으로 끝나는 부분Predictor SRAM 용량·레이턴시 파라미터를 새 캐시가 기존 L1/L2 계층과 다른 레이턴시로 갖도록 설정함. HBM 대역폭 파라미터를 KV 접근 baseline 설정에 사용함. 이는 .config 파일 수정으로 처리 가능함.
Layer 2 — GPGPU-Sim C++ 코드 수정 (핵심 기여)HMS의 HMSController.cc에 대응되는 구현으로 Predictor SRAM Controller와 KV Prefetch Scheduler 두 파일을 고려함.
Layer 3 — ISA Def 수정기존 LD/ST로 KV 접근을 모델링하는 한 초기 단계에서 불필요함. Prefetch를 memory controller가 투명하게 처리하면 기존 SASS trace 그대로 사용 가능함.
Layer 2 세부 구현
  1. Predictor SRAM Controller 신규 작성: Markov 테이블 조회 로직임. enqueue() 받아서 head_id → table hit/miss 판별 → tick에서 결과 반환함.
  2. KV Prefetch Scheduler (gmmu.cc 패턴 참고): 테이블 히트 시 다음 레이어의 KV를 미리 HBM에서 fetch함. HMS의 PCIE_VALIDATE 큐 패턴을 여기서 재활용할 수 있어 CPU-resident 데이터 fetch 경로가 이미 구현되어 있으니, 분기 조건만 head_id 기반으로 추가하면 됨.

4.2 [핵심] 구현의 방향성을 잡아보자

구현할 내용은 크게 세 개임.

대상평가 내용

KV prefetch controllerbandwidth 활용률, latency hiding, wasted prefetch

4.2.1 Head마다 달라지는 KV 이동 표현

  • ISA 수정 없이, 메모리 컨트롤러 레벨에서 mask로 처리하는 방식 생각중임.
  • 구체적으로?? KV cache에서 선택된 8개(K=8일 때 가정) head의 데이터에만 유효성 비트를 세우고, 나머지 8개 head의 fetch 요청을 컨트롤러 레벨에서 suppress함.
  • SASS trace에 있는 기존 LD 명령어를 그대로 쓰면서, memory controller가 실제로 HBM에 나가는 트래픽만 줄이는 효과를 만들 수 있지 않을까 싶음.

4.2.2 Predictor SRAM 모델링

  • 기존 L1 캐시 구조를 별도로 하나 더 만들고, 용량·레이턴시·포트 수만 SRAM 스펙에 맞게 Config에서 조정하는 방식 생각중임.
  • 구체적으로?? Accel-sim의 캐시 구조는 용량, associativity, hit/miss latency를 config 파라미터로 받음.
  • 여기에 “Predictor SRAM”이라는 이름의 새 캐시 인스턴스를 추가함.
  • 레이턴시는 SRAM 면적/전력을 먼저 추정하고, 거기서 나온 access latency를 config에 주입함. 어케 추정함?? 확인 필요함.
  • 테이블 lookup 요청이 SRAM 인스턴스에 먼저 들어오고, miss 시에만 DRAM으로 fallback하는 경로를 만듦. 실제로는 테이블이 SRAM에 완전히 올라간다고 가정하여 DRAM fallback은 발생 안 한다고 가정함.

4.2.3 Markov 조회와 KV Prefetch 연결

  • 조회 결과가 나오기 전에 prefetch를 언제 트리거하냐, 레이턴시 겹침을 어떻게 모델링할 것인가가 핵심임.
  • HMS의 “판단 → 큐잉 → tick 실행” 패턴을 그대로 써도 될 듯함.
  • 조회 요청을 큐에 넣는 순간 prefetch 큐도 동시에 세우는 방식 생각중임.
  • 구체적으로>?? 레이어 l의 attention 시작 시점에 SRAM 조회 요청을 enqueue함.
  • SRAM hit이 확인되는 순간(hit latency 후) KV prefetch 큐에 해당 head들의 HBM 주소를 넣음.
  • miss 시에는 원본 라우터 fallback 경로를 사용함 — 기존 HBM 접근 경로 그대로임.

1. 내가 HMS 논문에서 얻어갈 내용

HMS 코드 구조에서 HVRP Markov prefetch scheduler에 재사용할 구현 패턴 정리임

Study Note

재사용할 핵심 패턴

  • "판단 → 큐잉 → tick 실행" 패턴: 내 Markov prefetch scheduler도 이 패턴으로 구현하면 됨
  • post_process_cache_miss() → fill_pending queue → tick() 흐름을 그대로 재활용함
  • 판단: 요청이 들어온 시점에, 이 요청을 어떻게 처리할지 결정하는 단계임
  • 큐잉: 큐에 넣는 단계임
  • 실행: tick() 함수가 큐에서 요청을 꺼내서 실제로 처리하는 단계임

HMS 코드에서 배운 경로 판단

HMS의 첫 관문은 HMSController::enqueue()이고, 이 위치에서 DRAM / SCM / PCIe 경로를 판단한 뒤 tick()에서 실제 실행되는 구조임

내 상황의 대응Attention 계산 요청을 HVRPController::enqueue()에서 받아 head_id를 판별하고, SRAM/HBM 분기 후 tick()에서 prefetch 실행하는 방식으로 연결 가능함
HMS 논문 코드에서 배운 패턴
실제 GPU 요청
    ↓
HMSController::enqueue()   ← 첫 관문, 경로 판단
    ↓
[DRAM / SCM / PCIe] 분기
    ↓
tick()에서 실행            ← "판단 → 큐잉 → tick 실행" 패턴
HVRP에 재사용하는 패턴
Attention 계산 요청
    ↓
HVRPController::enqueue()  ← head_id 판별 + SRAM/HBM 분기
    ↓
[SRAM(on-chip table) / HBM(KV cache)] 분기
    ↓
tick()에서 prefetch 실행

PCIE_VALIDATE 큐

  • PCIE_VALIDATE 큐는 head를 fetch할 경로가 이미 구현되어 있음
  • 다만 실제로 HVRP에 그대로 활용 가능한지는 확인 필요함

Trace 재사용 전략

  • Trace 재사용 전략: HMS도 SASS trace 그대로 쓰고 page 메타데이터만 별도로 붙이는 방식임
  • 내 연구도 주소 → head_id 매핑 파일만 따로 만들면 trace 재생성 없이 가능함

Layer 2 — GPGPU-Sim C++ 코드 수정

  1. Predictor SRAM Controller 신규 작성 필요함. Markov 테이블 조회 로직이며, enqueue()를 받아서 head_id → table hit/miss 판별 후 tick에서 결과 반환함

Layer 3 — ISA Def 수정나의 경우는 기존 LD/ST로 KV 접근을 모델링하는 한 초기 단계에서 불필요함. Prefetch를 memory controller가 투명하게 처리하면 기존 SASS trace 그대로 사용 가능함

3.2.1 Head마다 달라지는 KV 이동을 시뮬레이터에서 어떻게 표현?

  • ISA 수정 없이, 메모리 컨트롤러 레벨에서 mask로 처리하는 방식 생각중임

구체적으로??

  • KV cache에서 선택된 8개(K=8일 때 가정) head의 데이터에만 유효성 비트를 세우고, 나머지 8개 head의 fetch 요청을 컨트롤러 레벨에서 suppress함
  • 이렇게 하면 SASS trace에 있는 기존 LD 명령어를 그대로 쓰면서, memory controller가 실제로 HBM에 나가는 트래픽만 줄이는 효과를 만들 수 있지 않을까 싶음

3.2.2 Predictor SRAM을 시뮬레이터에서 어떻게 모델링할 거냐?

  • 기존 L1 캐시 구조를 별도로 하나 더 만들고
  • 용량·레이턴시·포트 수만 SRAM 스펙에 맞게 Config에서 조정하는 방식 생각중임

구체적으로??

  • Accel-sim의 캐시 구조는 용량, associativity, hit/miss latency를 config 파라미터로 받음
  • 여기에 "Predictor SRAM"이라는 이름의 새 캐시 인스턴스를 추가함
  • 레이턴시는 SRAM 면적/전력을 먼저 추정하고, 거기서 나온 access latency를 config에 주입함. 어케 추정함?? 확인 필요함
  • 테이블 lookup 요청이 이 SRAM 인스턴스에 먼저 들어오고, miss 시에만 DRAM으로 fallback하는 경로를 만들기
  • 실제로는 DRAM fallback이 발생 안 한다고 가정함 — 테이블이 SRAM에 완전히 올라간다고 가정함

3.2.3 Markov 테이블 조회와 KV Prefetch를 어떻게 파이프라인으로 연결할 거냐?

조회 결과가 나오기 전에 prefetch를 언제 트리거하냐, 레이턴시 겹침을 어떻게 모델링할 것인가가 핵심임

  • HMS의 "판단 → 큐잉 → tick 실행" 패턴을 그대로 써도 될 듯함
  • 조회 요청을 큐에 넣는 순간 prefetch 큐도 동시에 세우는 방식 생각중임

구체적으로>??

  • 레이어 l의 attention 시작 시점에 SRAM 조회 요청을 enqueue함
  • SRAM hit이 확인되는 순간(hit latency 후) KV prefetch 큐에 해당 head들의 HBM 주소를 넣기
  • miss 시에는 원본 라우터 fallback 경로를 사용함 — 이건 기존 HBM 접근 경로 그대로임

4. 구현 위치

Predictor SRAM, UVM, KV Prefetch Scheduler를 시뮬레이터 계층에서 어디에 둘지에 대한 구조 정리임

Architecture Sketch

4.1 SRAM Cache

  • L2와 HBM 사이, 전용 on-chip SRAM으로 배치함
  • L1/Shared Memory는 SM마다 독립적이라 테이블 공유에 부적합함
  • L2는 용량이 부족함
  • 그렇다고 HBM에만 두면 SW lookup 문제가 생김

4.2 UVM

  • GPGPU-sim 자체에서 지원하지 않으나 간단한 수정으로 구현이 충분히 가능함
  • HMS도 그렇고 다른 사람들도 그렇고 각자 알아서 구현해서 쓰는 중임
  • 자세한 구현 방법은 타 논문 구현한 내용 확인해보고 구현해보는 게 좋겠음

4.3 Prefetch Scheduler

  • 레이어 l에서 Markov 테이블 조회 결과가 나오면,
  • 레이어 l+1의 attention에서 필요할 KV cache — 선택된 k개 head의 K, V — 를 미리 HBM에서 가져오는 것임
레이어 l 처리 중 KV prefetch 흐름

    ↓

    ↓

    ↓

HMS의 gmmu.cc 참고 방향

  • HMS의 gmmu.cc 파일 수정한 내용을 참조함
  • page fault 기반이 아니라 prediction 기반 prefetch라서 위치는 다름

Scheduler 배치 이유

  • prefetch 완료된 KV가 L2를 거쳐 SM에 올라오는 경로가 기존 demand fetch와 동일해서 추가 경로 불필요함
  • HMS의 HMSController::enqueue()가 정확히 이 위치에서 경로를 분기한 것과 동일한 패턴임
  • SM 레벨에 두면 SM마다 독립 스케줄러가 생겨서 중복 fetch 발생 가능함

6.0 고려해봐야하는 문제점

Prefetch timing, bandwidth contention, SRAM tag 확인 오버헤드에 대한 검증 필요 지점 정리임

추가 검증 필요

6.1 Prefetch 타이밍

  • 너무 일찍 하면 L2에서 evict되고 너무 늦으면 무의미함
  • 일찍 fetch해서 L2에서 evict되는 비율과 늦어서 demand와 겹치는 비율을 stat으로 뽑아서 최적 window를 찾는 실험 필요할 것으로 예상됨

6.2 Bandwidth 경쟁

  • Prefetch가 demand fetch랑 bandwidth 경쟁하면 오히려 느려질 수 있음
  • HMS의 Bypass Policy처럼, Fetch의 정책을 정확히 수립해놔야함
  • Prefetch가 항상 우선이고, demand fetch는 남은 bandwidth만 쓰는 구조임

6.3 SRAM tag 확인

  • SRAM Cache 구성 시 tag 확인하는 과정이 필요한지 확인 필요함
  • 이로 인해 생기는 오버헤드도 있을지 고려 필요함

Claude 왈 — Predictor SRAM의 특수성

  • 네 SRAM의 특수성 있음
  • 네 Predictor SRAM은 일반 캐시와 결정적으로 다른 점이 있음
  • 테이블의 key가 곧 주소야.
Predictor SRAM direct indexing 아이디어
조회 요청: (layer_idx, prev_8set, combo5)
               ↓
        uint32 packed key 생성
               ↓
        이 key로 SRAM 내 위치를 직접 계산 가능
남은 질문일반 캐시처럼 "임의의 메모리 주소가 어느 캐시 라인에 매핑되는지" 모르는 상황이 아님. key 자체가 구조화되어 있고, 테이블이 key 순으로 정렬되어 있음. 따라서 tag 확인을 일반 캐시처럼 모델링해야 하는지, direct-indexed table lookup으로 모델링해도 되는지 확인 필요함
